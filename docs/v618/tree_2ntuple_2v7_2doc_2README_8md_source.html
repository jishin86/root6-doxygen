1 RNTuple Introduction
2 ====================
3 
4 RNTuple (for n-tuple and nested tuple) is the experimental evolution of TTree columnar data storage. RNTuple introduces
5 new interfaces that aim to be more robust. In particular, the new interfaces are type-safe through the use of
6 templates, and the ownership is well-defined through the use of smart pointers. For instance
7 
8  tree->Branch("px", &Category, "px/F");
9 
10 becomes
11 
12  auto px = model->MakeField<float>("px");
13  // px is std::shared_ptr<float>
14 
15 The physical layout changes slightly from big endian to little endian so that it matches the in-memory layout on
16 most modern architectures. Combined with a clear separation of offset/index data and payload data for collections,
17 uncompressed RNTuple data can be directly mapped to memory without further copies.

20 Goals
21 -----
22 
23 RNTuple shall investigate improvements of the TTree I/O in the following ways
24 
25 1. More speed
26  * Improve mapping to vectorized and parallel hardware
27  * For types known at compile / JIT time: generate optimized code
28  * Optimized for simple types (float, int, and vectors of them)
29  * Better memory control: work with a fixed budget of pre-defined I/O buffers
30  * Naturally thread-safe and asynchronous interfaces
31 
32 2. More robust interfaces
33  * Compile-time type safety by default
34  * Decomposition into layers: logical layer, primitives layer, storage layer
35  * Separation of data model and live data
36  * Self-contained I/O code to support creation of a standalone I/O library

39 Concepts
40 --------
41 
42 At the **logical layer**, the user defines a data model using the RNTupleModel class.
43 The data model is a collection of serializable C++ types with associated names, similar to branches in a TTree.
44 The data model can contain (nested) collection, e.g., a type can be `std::vector<std::vector<float>>`.
45 
46 Each serializable type is represented by a **field**, concretely by a templated version of RField,
47 e.g. `RField<double>`. A field can generate or adopt an associated **value**, which represents a memory location
48 storing a value of the given C++ type. These distinguished memory locations are the destinations and sources for the
49 deserialization and serialization.
50 
51 The (de-)serialization is a mapping from the C++ type to the more simple **column** type system. A column contains
52 an arbitrary number of fixed-sized elements of a well-defined set of types: integers and floats of different
53 bit sizes. A C++ type may be mapped to multiple columns. For instance, an `std::vector<float>` maps to two columns,
54 an offset column indicating the size of the vector per entry, and a payload column with the float data.
55 
56 Columns are partitioned into **pages** (roughly: TTree baskets) of a few kB -- a few tens of kB each.
57 The **physical layer** (only) needs to provide the means to store and retrieve pages. The physical layer is
58 decoupled from the high-level C++ logic. The physical layer implements an abstract page storage interface,
59 so that dedicated implementations for key-value stores and other storage systems are conceivable.
60 At this point, the only provided backend stores the pages in ROOT files.
61 
62 NTuples are further grouped into **clusters**, which are, like TTree clusters, self-contained blocks of
63 consecutive entries. Clusters provide a unit of writing and will provide the means for parallel writing of data
64 in a future version of RNTuple.
